## 1. Sicherheits-Scan (Waechter)

### Hohe Befunde

| # | Befund | Datei | Empfehlung |
|---|--------|-------|------------|
| S-H1 | **Kein Account-Lockout** – Rate Limiting existiert (20/15min), aber kein Account-spezifischer Lockout. Brute-Force ueber verschiedene IPs moeglich. | `backend/routes/auth.js` | Account-Lockout nach 5 Fehlversuchen (15min Sperre). Zaehler in DB `login_attempts`. |
| S-H2 | **Fehlende Input-Validierung** – Keine systematische Schema-Validierung (Zod/Joi) fuer Request-Bodies. Verlaesst sich auf DB-Constraints. | Diverse Route-Handler | Zod-Schema-Validierung als Middleware fuer alle Public-Endpunkte. |
| S-H3 | **`dangerouslySetInnerHTML`** in E-Mail-Vorschau – Stored-XSS-Risiko bei kompromittiertem Admin-Account. | `EmailBrandingTab.tsx` | DOMPurify vor dem Rendering oder Vorschau in `<iframe sandbox>`. |
| S-H4 | **Logout: clearCookie ohne vollstaendige Cookie-Optionen** – Nur `{ path: '/' }` statt `httpOnly`, `secure`, `sameSite`. Browser loeschen den Cookie ggf. nicht zuverlaessig. | `backend/routes/auth.js:132-146` | `clearCookie('token', cookieOptions())` mit allen Attributen. |
| S-H5 | **Keine serverseitige Token-Revocation** – Logout loescht nur den Cookie clientseitig. Ein gestohlenes JWT bleibt 8h gueltig. | `backend/routes/auth.js` | Redis-Blacklist mit JWT-`jti` oder `invalidated_before` pro User. |

### Mittlere Befunde

| # | Befund | Empfehlung |
|---|--------|------------|
| S-M1 | SameSite=Lax statt Strict | Bewusste Designentscheidung, dokumentiert |
| S-M2 | Kein Passwort-Komplexitaets-Check (nur min. 8 Zeichen) | Min. 10 Zeichen + Grossbuchstabe + Zahl |
| S-M3 | Keine Request-ID/Correlation-ID in Logs | UUID-Middleware + Pino-Integration |
| S-M4 | Docker-Container laeuft als Root | `USER node` in Dockerfile.backend |
| S-M5 | PostgreSQL-Port exponiert in docker-compose.yml | Port-Mapping fuer Prod entfernen |
| S-M6 | Keine Dependency-Vulnerability-Checks in CI/CD | `npm audit` als CI-Job |
| S-M7 | CSRF nur via SameSite+CORS (bewusst akzeptiert) | Dokumentiert, akzeptables Restrisiko |
| S-M8 | SSW/BL Admin-Routes ohne Rate-Limiter – `/api/bl/admin` und `/api/ssw/admin` ohne Rate-Limit (nur Auth) | `rateLimiters.admin` beim Mount ergaenzen |
| S-M9 | CSV-Upload ohne fileFilter – Multer akzeptiert beliebige MIME-Typen fuer "CSV"-Import | `fileFilter` fuer `text/csv`, `text/plain`, Endung `.csv` |

### Niedrige Befunde

| # | Befund |
|---|--------|
| S-N1 | JWT-Secret als Env-Variable statt Docker Secret |
| S-N2 | Kein Refresh-Token (8h Lebensdauer) |
| S-N3 | Kein API-Versioning |
| S-N4 | `error.message` in einigen 500er-Responses |
| S-N5 | HSTS ohne preload/includeSubDomains |
| S-N6 | LIMIT als Template-Literal in dataSubject.js:522 (nicht nutzergesteuert, aber Style-Verstoss) |
| S-N7 | booking_requests SELECT mit LIMIT 3000 ohne Pagination (teacher.js:379) |

### Positiv-Befunde

- SQL-Injection: Alle Queries parametrisiert
- XSS: React-Escaping + CSP + E-Mail-Escaping
- Auth: JWT httpOnly Cookies, kein Bearer-Token
- CORS: Strikte Origin-Whitelist
- Rate Limiting: Auf allen relevanten Endpunkten
- File Upload: MIME-Check, Extension-Whitelist, SVG blockiert
- Security Headers: Helmet korrekt (CSP, X-Content-Type, X-Frame)
- RLS: Auf kritischen Tabellen aktiviert
- Audit Logging: PII-Zugriffe und Security-Events geloggt
- Passwort-Hashing: bcrypt, 10 Runden
- Keine Secrets in Versionskontrolle
